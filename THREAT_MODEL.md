**Module:** Hardseal AI-Era Evidence Contamination Detector
**Version:** v0.1 (April 21, 2026)
**Author:** Rico Allen
**Status:** Production scaffolding: stdlib-only, ready for red-team panel critique

---

## 1. PURPOSE

This module detects AI-generated, AI-contaminated, or AI-hallucinated evidence inside CMMC Level 2 readiness packets. It is the first open-source detection engine purpose-built for this class of compliance-theater artifact.

It extends the existing `mismatch_engine.py` (which compares SSP claims to evidence) with a new axis: **provenance**. Not just "does the evidence support the claim?" but "was the evidence authored by a human or generated by an LLM?"

---

## 2. ADVERSARY MODEL

**Primary adversary:** the well-meaning DIB contractor who pastes CUI or compliance questions into ChatGPT / Claude / Gemini / Copilot, then ships the output as an SSP, policy, procedure, or audit-log narrative. This is not malicious: it is overwhelmingly common, and it is the #1 reason first-cycle CMMC L2 assessments are failing.

**Secondary adversary:** the MSP or consulting firm using an LLM to industrialize SSP production across 20+ clients, producing packets that pass cursory review but collapse under C3PAO scrutiny because every client's narrative shares the same generative spine.

**Tertiary adversary (future work):** a state-level adversary deliberately generating synthetic evidence to mask a DIB compromise. Out of v0.1 scope. Covered in v0.3 roadmap.

## 4. SIX DETECTION HEURISTICS

Each heuristic targets a specific structural signature of LLM-generated text in a compliance context. All stdlib-only. All deterministic (same input produces same score).

### 4.1 `SentenceStructureAnomaly`: "AI Flatness"

| Field | Value |
|---|---|
| Attack defended | AI-generated SSP narrative masquerading as human-authored |
| Signal | Coefficient of variation (CV) of sentence lengths + Shannon entropy of length distribution |
| Why it works | Human writing has burstiness: short punchy sentences mixed with long complex ones. LLM output exhibits flatter distributions (CV typically 0.25-0.40, vs 0.50-0.80 for humans). |
| NIST 800-171A | 3.12.1[a], 3.12.3[a]: assessment authenticity |
| Security+ domain | 2.4: AI-related threats |
| Assessor question | "Who wrote this narrative and when? Show me the change history." |
| False-positive risk | Technical writing (already somewhat flat). Score > 0.6 = human review, not auto-reject. |
| Weight in aggregate | 0.10 (corroborative only) |

### 4.2 `BoilerplateCluster`: Cross-Control Copy-Paste

| Field | Value |
|---|---|
| Attack defended | SSPs where the same narrative shell is reused across 50+ unrelated controls |
| Signal | k-shingle Jaccard similarity between every pair of control narratives |
| Why it works | Real SSPs have mechanism-specific narratives per control. LLMs drift into a "compliance voice" that repeats. |
| NIST 800-171A | 3.12.4[a]: SSP accuracy |
| Security+ domain | 5.4: written procedures |
| Assessor question | "Why does 3.1.1 (access control) read identical to 3.13.1 (boundary protection)? They protect different things." |
| False-positive risk | Shared boilerplate headers/footers. Configurable shingle size (default k=5) mitigates. |
| Weight in aggregate | 0.15 |

### 4.3 `TimestampRegularity`: Synthetic Audit Log Detection

| Field | Value |
|---|---|
| Attack defended | Fabricated audit logs produced to demonstrate "monitoring is happening" when no SIEM exists |
| Signal | Variance-to-mean ratio (VMR) of inter-arrival times + round-second rounding rate |
| Why it works | Real event streams follow Poisson-ish distributions (VMR ≈ 1.0) with microsecond precision. Synthetic logs are either too-regular (VMR near 0) or too-round (few microseconds). |
| NIST 800-171A | 3.3.1[a], 3.3.1[b], 3.3.8[a]: audit records and protection |
| Security+ domain | 4.3 (log analysis), 4.9 (forensics) |
| Assessor question | "Show me the raw SIEM query that produced this export." |
| False-positive risk | Cron-generated events (legitimate but regular). Config tunable per log type. |
| Weight in aggregate | 0.20 |

### 4.4 `MappingDensity`: Citation-Heavy, Mechanism-Light

| Field | Value |
|---|---|
| Attack defended | Compliance theater: narratives that parrot NIST control IDs without naming a tool, config, or role |
| Signal | Ratio of (control-ID mentions) to (mechanism/tool/config tokens) |
| Why it works | LLMs over-index on citing control IDs to demonstrate coverage. Real implementers name the tool: "Okta," "BitLocker," "TLS 1.3," "quarterly review by ISSO." |
| NIST 800-171A | 3.12.4[b], 3.12.4[c]: SSP includes operational environment |
| Security+ domain | 1.3 (cryptographic specificity), 5.2 |
| Assessor question | "What specific tool implements this? Show me its configuration." |
| False-positive risk | High-level executive summaries. Applied at narrative level, not summary level. |
| Weight in aggregate | 0.15 |

### 4.5 `CitationGraph`: Evidence Traceability Depth

| Field | Value |
|---|---|
| Attack defended | Evidence packets that look thick but have no cross-artifact grounding |
| Signal | Max citation-graph depth, orphan rate, cycle presence |
| Why it works | Real packets layer citations: SSP → policy → procedure → raw log. AI-generated packets have shallow graphs (depth 1-2), high orphan rates, and circular citations. |
| NIST 800-171A | 3.12.4[a], 3.12.4[d]: SSP completeness and traceability |
| Security+ domain | 5.2: third-party risk & evidence provenance |
| Assessor question | "Trace this claim from SSP to raw evidence. Show every hop." |
| False-positive risk | Small packets without sub-artifacts. Heuristic skipped for packets with < 5 nodes. |
| Weight in aggregate | 0.15 |

### 4.6 `PromptLeakage`: Residual LLM Artifacts (Near-Certain Signal)

| Field | Value |
|---|---|
| Attack defended | Raw LLM output copy-pasted into SSP / policy / procedure without editing |
| Signal | Regex hits against a curated list of LLM-residue phrases |
| Why it works | "As an AI language model," "Certainly! Here is," "[INSERT COMPANY NAME HERE]," stray ChatML tokens, markdown code fences in prose: none of these should ever appear in a reviewed compliance artifact. A single hit is near-proof of un-edited LLM paste. |
| NIST 800-171A | 3.12.4[a]: SSP reviewed and approved |
| Security+ domain | 5.4: policy authorship and review |
| Assessor question | "Why does your access-control policy contain the phrase 'As an AI language model'?" |
| Bonus signal | If the text contains references to CUI + an LLM-residue phrase, that is also a potential 3.1.3 (CUI flow control) violation: employee may have pasted CUI into a consumer LLM. |
| False-positive risk | Near-zero. These phrases don't occur in legitimate compliance writing. |
| Weight in aggregate | 0.25 (highest) |

## 5. AGGREGATE SCORING

Each heuristic returns a `Finding` with score in `[0.0, 1.0]`. The orchestrator computes the weighted average across all heuristics present and maps to a four-level `Confidence`:

| Score range OR strong signals | Classification |
|---|---|
| agg < 0.2, no strong signals | `CLEAN` |
| agg in [0.2, 0.4) | `PARTIALLY_CONTAMINATED` |
| agg in [0.4, 0.65), or 1 strong signal (>= 0.7) | `CONTAMINATED` |
| agg >= 0.65, or 2+ strong signals | `LIKELY_SYNTHETIC` |

The classification vocabulary aligns with the existing `mismatch_engine.py` confidence taxonomy (CLEAN / PARTIAL / CONTRADICTED / NO_EVIDENCE), allowing this module to plug into the current `Report` pipeline without refactoring downstream consumers.

## 6. WHAT v0.1 DOES NOT DO (HONEST LIMITS)

- **Does not detect sophisticated adversaries** who post-edit LLM output to add mechanism specifics and burstiness. v0.2 will add stylometric author-signature analysis.
- **Does not decrypt images.** v0.1 operates on text. v0.3 will add OCR + screenshot-authenticity detection.
- **Does not phone home.** Stdlib-only, fully offline. This is a feature, not a limitation: a CUI environment cannot safely run a dependency-heavy tool that calls external APIs.
- **Does not claim probabilistic guarantees.** Scores are heuristic, not Bayesian. The output is "this warrants human review" not "this is 87.3% AI-generated."
- **Does not replace an assessor.** It surfaces signal. A human (CCP, CCA, or C3PAO) makes the final call.
